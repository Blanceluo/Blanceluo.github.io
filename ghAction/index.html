GitHub Actions 持续集成部署

12-8-2020

前言

本篇文章，是由作者在搭建博客时，略微涉及 GitHub Actions 的自动化持续集成部署后写下的。 因此，您在阅读时，可能会发现诸多错误。如果存在，请与作者联系，十分感谢。

关于更多，您可以参考 GitHub Actions 文档。

此文章面向的是假定有一定计算机基础，了解某种或多种计算机语言，甚至掌握一些服务端环境配置经验的人群。

什么是 自动化持续集成

一个项目，在开发阶段结束后，进入测试阶段时。 一般都会频繁的进行编译打包，将项目文件持续地发布更新到测试环境中去等操作。 类似于这种重复性质较高的工作，人是情绪化的动物，超过一定的次数难免惹人焦躁。情绪生变，工作便容易出错。

因此，若将这类性质的工作，交由某些服务或工具执行。而开发人员只需编辑一些相应的文件配置。如此这般，不仅简化了操作流程，对开发人员的体验更加友好。同时也大大降低了出错几率，工作质量也能得到一定的保证。

当某个需求痛点的出现，定会催生出相应的解决途径。基于此，业界早已推出了一些成熟的解决方案。 如 Jenkins, Travis CI, Netlify, GitHub Actions…

作者在搭建个人博客时，对比了一些常见的博客框架。 Hexo，Hugo，VuePress 乃至于作者最终选择的 Gatsby。对于一篇博文的编辑，发布都需要经历以下几个流程。

新建博文 —> 编译打包源码 —> 发布上线

久而久之，不胜其烦。身为一位计算机行业编码人员，已深谙封装复用之便利，如何能够受得了这般的重复繁琐。加之作者博客源码托管在 GitHub 上，自然选择 GitHub Actions。

此为作者的配置文件，敏感处已隐去。特加入中文注释说明。

# GitHub Actions 配置文件以 YML 编写。

# 工作流程名称 可选。若设置，将会出现在仓库的 Actions 下拉选项卡中。
name: Gatsby Publish

on: # 指定触发工作流程文件的事件，此处为 push 事件。
  push:
    branches: # 指定触发工作流程文件的分支，此处为 master 分支。
      - master

jobs: # 一系列作业的组合
  build: # build 作业
    runs-on: ubuntu-latest # 执行作业的虚拟环境
    steps: # 作业中的所有步骤
      # uses 关键字将会使用社区中已封装好的操作

      # 该操作将会检出仓库且将其下载到运行环境中
      - uses: actions/checkout@v1
      # 该操作将会在运行环境中安装 Node 软件包。用以下方执行 NPM 命令
      - uses: actions/setup-node@v1
        with:
          node-version: "12.x" # 指定 Node 版本

      # run 关键字将会在运行环境中执行特定命令

      - run: npm i  # 执行 npm i，安装依赖
      - run: npm run build # 执行打包编译

      # 该操作会将特定目录下的文件 push 到指定仓库的指定分支
      # 作者这里创建了另外一个仓库存储编译后的项目文件
      - uses: peaceiris/actions-gh-pages@v2.5.1
        env:
          # secrets.ACCESS_TOKEN
          # 加密环境变量，用以虚拟环境下访问作者仓库
          ACTIONS_DEPLOY_KEY: ${{ secrets.ACCESS_TOKEN }}
          EXTERNAL_REPOSITORY: REPOSITORY # 指定被 push 的仓库名
          PUBLISH_BRANCH: master # 指定被 push 的分支
          PUBLISH_DIR: ./public # 指定将要 push 的目录

到这里，只是实现了 自动 打包编译源码 的步骤。还剩下将编译后的文件自动发布到线上服务器。

其实，这最后一步，无非便是实现 线上服务器特定目录能够在所对应的远程仓库文件发生变更后自动拉取到本地。我们可以借助 GitHub webhooks 实现.

GitHub webhooks

只要在仓库或组织上执行特定的操作，便可触发。例如，您可以配置 webhooks 在以下情况下执行：

1. 推送到仓库
2. 打开 pull request
3. 构建 GitHub Pages 页面
4. 添加新成员到团队

所以我们自动部署的实现步骤主要是：

1. 推送到仓库 (已实现)
2. 触发 Hooks，发送指定请求到服务器
3. 服务器接收到指定请求，执行自动部署脚本
4. 自动部署脚本执行代码拉取，移动文件等完成网站更新部署的操作。

2. 编写服务脚本

作者在这里使用 Node 编写，当然，使用任意一种服务端语言皆可。

// server.js

const http = require('http')
const createHandler = require('github-webhook-handler')

// secret 与 GitHub Hooks 后台设置保持一致
const handler = createHandler({ path: '/', secret: 'xxx' })

function run_cmd(cmd, args, callback) {
  const spawn = require('child_process').spawn;
  const child = spawn(cmd, args);
  const resp = "";
  child.stdout.on('data', function(buffer) { resp += buffer.toString(); });
  child.stdout.on('end', function() { callback (resp) });
}

http.createServer(function (req, res) {
  handler(req, res, function (err) {
    res.statusCode = 404
    res.end('no such location')
  })
}).listen(7000)

handler.on('error', function (err) {
  console.error('Error:', err.message)
})

handler.on('push', function (event) {
  console.log('Received a push event for %s to %s',
    event.payload.repository.name,
    event.payload.ref);

    // 执行自动部署脚本
    run_cmd('sh', ['./deploy.sh', event.payload.repository.name], function(text){ console.log(text) });
